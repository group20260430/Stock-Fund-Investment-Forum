## 三、Git/GitHub 协作规范

### 3.1 分支策略

```
main              ← 生产分支（禁止直接推送）
  └─ develop      ← 开发主分支（贺嘉轩 Merge）
       ├─ feat/backend/user-api        ← 陶畅
       ├─ feat/backend/content-api     ← 张照炎
       ├─ feat/backend/social-api      ← 杨文弢
       ├─ feat/frontend/core           ← 张桐尘
       ├─ feat/frontend/pages          ← 刘嘉成
       └─ docs/xxx                     ← 文档分支
```

### 3.2 分支命名规范

| 类型 | 格式 | 示例 |
|------|------|------|
| 功能开发 | `feat/模块/功能名` | `feat/backend/user-register` |
| Bug修复 | `fix/模块/问题描述` | `fix/frontend/login-error` |
| 文档 | `docs/文档名` | `docs/api-design` |
| 重构 | `refactor/模块/描述` | `refactor/backend/auth` |

### 3.3 提交信息规范

采用 **Conventional Commits** 格式：

```
<类型>(<范围>): <简短描述>

[可选正文]
```

**示例：**
```
feat(backend): 实现手机号注册接口
fix(frontend): 修复登录页验证码倒计时不刷新问题
docs: 更新 API 接口文档
refactor(backend): 重构 JWT 鉴权中间件
```

**类型选项：** `feat`、`fix`、`docs`、`refactor`、`test`、`chore`

### 3.4 Pull Request 流程

```
1. 从 develop 创建功能分支
        ↓
2. 本地开发 + 自测
        ↓
3. 推送远程分支，创建 PR 到 develop
        ↓
4. 贺嘉轩 或 另一名组员 Code Review
        ↓
5. Review 通过 → Squash Merge
   Review 不通过 → 修改后重新提交
```

**PR 模板：**
```markdown
## 描述
简要描述本次改动内容

## 关联需求
相关用户故事/Issue 编号

## 改动清单
- [ ] 功能 A
- [ ] 功能 B

## 测试情况
- [ ] 本地测试通过
- [ ] 接口测试通过

## 截图（UI 改动时提供）
```

## 四、代码规范

### 4.1 通用规范

- **缩进：** 统一使用 4 个空格（非 Tab）
- **编码：** UTF-8
- **命名：** 语义化命名，禁止使用拼音缩写
- **注释：** 关键逻辑必须写注释，复杂函数写文档注释
- **行尾：** LF（Unix 风格），禁止使用 CRLF

### 4.2 后端规范（Python FastAPI）

```python
# 文件命名：snake_case.py
# 函数命名：snake_case
# 类命名：PascalCase
# 变量命名：snake_case
# 常量命名：UPPER_SNAKE_CASE

# 示例
class UserService:
    """用户服务类 - 处理用户相关的业务逻辑"""
    
    async def register_user(self, phone: str, password: str) -> User:
        """用户注册"""
        pass

MAX_LOGIN_ATTEMPTS = 5
```

### 4.3 前端规范（Vue 3 + JavaScript）

```javascript
// 文件命名：PascalCase.vue (组件) / camelCase.js (工具)
// 组件命名：PascalCase
// 函数命名：camelCase
// 变量命名：camelCase
// CSS 类名：kebab-case

// 示例
// components/UserCard.vue
// utils/formatDate.js

const userList = ref([]);
function fetchUserData() { }
```

### 4.4 数据库规范

- **表名：** 小写复数，`snake_case`（如 `users`、`posts`）
- **字段名：** 小写 `snake_case`
- **主键：** `id`（自增整数或 UUID）
- **外键：** `被引表名_singular_id`（如 `user_id`）
- **时间字段：** `created_at`、`updated_at`
- **软删除：** `deleted_at`（可为空，非空表示已删除）

## 五、开发流程规范

### 5.1 每日开发流程

```
① 拉取最新 develop 代码
        ↓
② 在功能分支上开发
        ↓
③ 完成功能 → 本地测试
        ↓
④ 提交代码 → 推送远程
        ↓
⑤ 创建 PR → 请求 Review
        ↓
⑥ Review 通过 → 合并到 develop
```

### 5.2 接口对接流程

```
前端组 ←→ 贺嘉轩 ←→ 后端组
    ↑                      ↑
    └──────────────────────┘
         (通过 API 文档对齐)
```

1. 后端组先在 `backend_api.md` 中定义接口（路径、请求/响应格式）
2. 贺嘉轩审查接口设计是否合理
3. 前端组按接口文档开发前端页面
4. 前后端联调时，贺嘉轩协调接口变更

### 5.3 代码审查要点

| 检查项 | 说明 |
|--------|------|
| 功能正确性 | 是否实现了需求中描述的功能 |
| 异常处理 | 是否有 try-catch，边界情况是否处理 |
| 安全性 | 是否存在 SQL 注入、XSS、CSRF 等风险 |
| 性能 | 是否存在 N+1 查询、不必要的循环 |
| 代码风格 | 是否符合组内规范 |
| 注释质量 | 复杂逻辑是否有注释，注释是否准确 |
| 测试覆盖 | 关键函数是否有对应测试 |

3. **代码相关提问** → 附上代码片段和错误信息
4. **接口变更** → 必须在群里通知相关方
5. **每日更新** → 在群里简要说一下"今天做了什么、遇到什么问题"

## 九、技术选型（初步）
